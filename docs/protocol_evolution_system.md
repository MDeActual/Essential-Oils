# protocol_evolution_system.md — Protocol Evolution System Design

## Purpose

This document specifies the system design for protocol evolution in the Phyto.ai platform. It defines how protocols are versioned, how evolution signals are generated and evaluated, and how new protocol versions are produced and deployed.

This document covers **system design and data structures**. For the governance workflow (approval gates, rollback procedures, human-in-the-loop rules), see `docs/autonomous_iteration_protocol.md`.

**Moat boundary**: The population analytics signal model and the protocol generation algorithm are protected per M-002 and M-004 in `docs/MOAT_MODEL.md`.

---

## Core Concepts

### Protocol Version

A protocol version is an immutable snapshot of a protocol's phase structure, challenge set, oil assignments, and duration. Versions follow semantic versioning per LOCK-005:

| Bump Type | When Applied |
|-----------|-------------|
| `MAJOR` | Fundamental restructuring of protocol goals or phase count |
| `MINOR` | Phase modification, challenge update, oil substitution |
| `PATCH` | Documentation correction, metadata update only |

Each version is stored and never deleted. Deprecation is managed via the `status` field on the Protocol entity.

---

### Evolution Signal

An `EvolutionSignal` is the typed output of the Contributor Analytics Agent when a trigger condition is met (defined in `docs/autonomous_iteration_protocol.md`). It initiates the evolution workflow.

```json
{
  "signal_id": "sig-<uuid>",
  "trigger": "low_adherence | high_challenge_skip_rate | outcome_plateau",
  "protocol_id": "<uuid>",
  "protocol_version": "<semver>",
  "evidence_count": "<integer>",
  "metric": "<metric_name>",
  "metric_value": "<float>",
  "timestamp": "<iso8601>"
}
```

**Hard gate**: `evidence_count` must be ≥ 30 `real_contributor` records. Signals with fewer records must not be emitted (enforced by LOCK-003).

---

### Iteration Candidate

An `IterationCandidate` is generated by the Protocol Evolution Agent in response to an `EvolutionSignal`. It proposes a specific, scoped change.

```json
{
  "candidate_id": "cand-<uuid>",
  "signal_id": "<signal_id>",
  "type": "phase_modification | challenge_update | oil_substitution | duration_adjustment",
  "proposed_change": "<human-readable description>",
  "rationale": "<evidence-based rationale citing signal data>",
  "risk_level": "low | medium | high",
  "requires_human_approval": "<boolean>",
  "affected_protocol_id": "<uuid>",
  "target_version_bump": "MAJOR | MINOR | PATCH"
}
```

**Auto-approval rules (Swarm Orchestrator only)**:
- `low` or `medium` risk + non-structural type = may proceed to shadow deployment without human review
- `high` risk or `phase_modification` type = requires human approval before any action
- Maximum one auto-approved evolution cycle per protocol version without cumulative human review

## System Components

### 1. Analytics Aggregation Layer (`src/analytics/`)

Responsibilities:
- Reads Contributor Records from the data store
- Filters for `data_origin: real_contributor` and `exclusion_status: included`
- Computes per-protocol metrics: average adherence, challenge skip rates, outcome improvement rates
- Monitors for trigger threshold crossings
- Emits `EvolutionSignal` objects to the Swarm Orchestrator when thresholds are met

**Outputs**: `EvolutionSignal` objects (routed through Swarm Orchestrator)

---

### 2. Candidate Generation Layer (`src/protocol/evolution/`)

Responsibilities:
- Receives `EvolutionSignal` from Swarm Orchestrator
- Analyzes the affected protocol phase, challenge, or oil
- Generates 1–3 `IterationCandidate` objects
- Assigns `risk_level` and `requires_human_approval` based on candidate type and scope
- Routes candidates back to Swarm Orchestrator for review gate evaluation

**Outputs**: `IterationCandidate` objects

---

### 3. Review Gate (`src/protocol/evolution/`)

Responsibilities:
- Evaluates each candidate against risk and type criteria
- Routes low/medium non-structural candidates to shadow deployment
- Holds high-risk or structural candidates in a review queue for human approval
- Enforces the one auto-cycle-per-version guard

---

### 4. Shadow Deployment Layer (`src/protocol/`)

Responsibilities:
- Creates a new protocol version (version bump applied)
- Assigns the new version to a shadow cohort (≥ 10 contributors)
- Monitors shadow cohort adherence and outcome metrics for 14 days
- Compares shadow cohort performance to baseline

**Promotion criteria**: Shadow cohort adherence ≥ (baseline adherence − 5%)
**Rollback criteria**: Shadow cohort adherence < (baseline adherence − 5%)

---

### 5. Protocol Version Store (`src/protocol/`)

Responsibilities:
- Maintains the immutable version history for each protocol
- Provides rollback capability to any previous version
- Exposes version metadata to the protocol generation engine for routing

## Evolution Metrics

The following metrics are tracked per protocol version for signal generation and shadow cohort evaluation:

| Metric | Description | Unit |
|--------|-------------|------|
| `avg_adherence` | Average adherence across active contributors | % |
| `challenge_skip_rate` | Per-challenge skip rate across contributors | % |
| `outcome_improvement_rate` | % of contributors reporting improvement | % |
| `completion_rate` | % of contributors completing the protocol | % |
| `protocol_score` | Composite score from protocol scoring system | 0–100 |
| `evidence_count` | Number of eligible Contributor Records | integer |

---

## Protocol Score Formula

The protocol score is a composite metric used for ranking, recommendation, and evolution signals:

```
Protocol Score =
  (Outcome Improvement Rate × 0.40) +
  (Adherence Rate × 0.30) +
  (Completion Rate × 0.20) +
  (Sample Size Weight × 0.10)
```

Where `Sample Size Weight` is a confidence factor that scales with evidence count toward a cap at 100 eligible records.

**Integrity filters applied before scoring**:
- Exclude records with `data_origin ≠ real_contributor`
- Exclude records with `adherence_score < 50`
- Exclude records with flagged anomalies

---

## Rollback Procedure

If shadow or promoted deployment causes adherence regression > 10% relative to baseline:

1. Revert the protocol to the previous `active` version immediately.
2. Emit an `IterationRollback` event:

```json
{
  "rollback_id": "rb-<uuid>",
  "candidate_id": "<candidate_id>",
  "protocol_id": "<protocol_id>",
  "reverted_to_version": "<semver>",
  "regression_metric": "avg_adherence",
  "regression_value": "<float>",
  "baseline_value": "<float>",
  "timestamp": "<iso8601>"
}
```

3. Mark the candidate as `REJECTED` in the log.
4. Append an entry to `docs/ARCHITECTURE_DECISION_LOG.md` if structural.
5. Alert the human project lead.

## Evolution Log Format

Every evolution cycle must produce a log entry:

```json
{
  "evolution_log_id": "elog-<uuid>",
  "signal_id": "<signal_id>",
  "candidate_id": "<candidate_id>",
  "protocol_id": "<protocol_id>",
  "from_version": "<semver>",
  "to_version": "<semver>",
  "status": "ACCEPTED | REJECTED | PENDING",
  "approved_by": "auto | human:<identifier>",
  "shadow_cohort_adherence": "<float>",
  "baseline_adherence": "<float>",
  "timestamp": "<iso8601>"
}
```

---

## Constraints

- Agents may never modify locked architecture documents through this workflow.
- Simulation data must never be used as the evidence base for a real evolution signal.
- The 30-record minimum evidence count is a hard gate enforced by LOCK-003 and may not be waived by any agent.
- Protocol evolution is disabled by default at staging via the `ENABLE_PROTOCOL_EVOLUTION=false` feature flag.
