# swarm_rules.md — Data Integrity and Multi-Agent Execution Rules

## Purpose

This document defines the fixed governance rules and operating constraints that govern agent swarm behavior in the Phyto.ai protocol intelligence platform. All agents, automated pipelines, and developer workflows must comply with these rules.

---

## 1. Product Identity Rules

Build Phyto.ai as:
- a community-driven AI-native natural health discovery platform grounded in essential oil therapeutics

Do not build it as:
- a generic wellness tracker
- a supplement-first storefront
- an open unmoderated community forum
- a thin AI wrapper with little product depth

---

## 2. Build Priority Rules

Always prioritize in this order:

1. Trust and privacy
2. Data integrity
3. Protocol engine
4. Challenge engine
5. Contributor workflows
6. Discovery outputs
7. Reputation and quality systems
8. Growth systems
9. Monetization layers

Do not prioritize:
- Advanced monetization before user value
- Open social features before moderation and reputation systems exist
- External integrations before core workflows are stable
- Protocol evolution before sufficient real contributor data exists (minimum 30 records)

---

## 3. Documentation Rules

- Mandatory first read before any implementation: `CLAUDE.md` → `docs/ARCHITECTURE_INDEX.md`
- Use latest version of documents only
- Do not implement from mixed versions of the same file family
- Update authoritative docs when architecture-level changes are introduced
- Do not create duplicate canonical files for the same subsystem
- All new files must be registered in `docs/ARCHITECTURE_INDEX.md`

---

## 4. Data Integrity Rules

Every analytics-eligible contributor-linked record must include:
- `data_origin`
- `exclusion_status`

### Allowed `data_origin` values

| Value | Description |
|-------|-------------|
| `real_contributor` | Actual platform participant |
| `synthetic_simulation` | Generated by the simulation layer |
| `internal_test` | Internal QA or test data |

### Allowed `exclusion_status` values

| Value | Description |
|-------|-------------|
| `included` | Eligible for production analytics |
| `excluded` | Not eligible for production analytics |

### Eligibility Rules

Only `real_contributor` data may appear in:
- Production discovery feed
- Public challenge result summaries
- Public protocol scores
- Research exports labeled as real-world evidence

### Adherence Handling

| Adherence Range | Treatment |
|-----------------|-----------|
| `< 50%` | Exclude from scoring; `exclusion_status: excluded` |
| `50–69%` | Down-weight; low confidence flag |
| `≥ 70%` | Full weight |

---

## 5. Privacy Rules

Treat symptom, protocol, outcome, and location-related data as sensitive.

- Collect the minimum necessary data
- Use anonymized aggregated statistics for model learning and discovery outputs
- Never persist raw GPS coordinates into discovery-facing analytics
- Location data must be opt-in
- Clearly explain data use at challenge join and relevant product moments

Trust wins over speed.

---

## 6. Protocol Rules

Never show vague protocol steps to users.

Every user-facing protocol step must resolve into:
- Method (aromatic, topical, internal)
- Timing (morning, evening, frequency)
- Quantity or intensity (drops, dilution ratio)
- Safety note if required

Protocol suggestions must flow through:

```
protocol definition
→ protocol engine
→ instruction layer
→ user display
```

Do not allow free-form protocol claims to bypass structured protocol review.

---

## 7. Engineering Rules

- Use typed contracts for all inter-agent communication
- Use migration-based schema evolution
- Use feature flags for risky features
- Provide deterministic fallbacks when AI is unavailable
- Log all critical actions (see list below)
- No hidden production shortcuts
- No debug bypasses in production code paths

### Critical events to log

- Protocol generation
- Challenge joins
- Challenge completion
- Outcome submission
- Reputation recalculation
- Exclusion validation
- Evolution signal emission
- Iteration approval or rejection

---

## 8. Testing Rules

Before any release, validate:

- Synthetic data is excluded from production analytics
- Challenge result integrity is maintained
- Protocol scoring integrity passes
- Adherence exclusion thresholds are enforced
- Reputation weighting behavior is correct

### Minimum acceptance journey

1. User joins challenge
2. User follows protocol
3. User logs daily signals
4. Completion is calculated
5. Result summary is generated without exposing protected internals

---

## 9. Agent Authority Rules

- All agents operate within the authority matrix defined in `AGENTS.md`
- No agent may expand its own authority level
- All inter-agent coordination routes through the Swarm Orchestrator
- Conflict resolution precedence: Human Lead > Swarm Orchestrator > Domain Agents

---

## 10. Moat Integrity Rules

The following must never appear in external-facing outputs, API responses, or public documentation:

- Synergy scoring matrix weights or computation method (M-001)
- Protocol generation intermediate states or algorithm (M-002)
- Challenge engine rule configuration (M-003)
- Population analytics signal extraction methodology (M-004)
- Full oil ontology graph or relational structure (M-005)

See `docs/MOAT_MODEL.md` for the complete boundary definition.

---

## 11. Meta-Rule

**No implementation without dependency confirmation.**

Before any subsystem work begins, verify:

- Authoritative documentation exists for the subsystem
- Upstream dependencies are implemented or explicitly scoped
- Schema impact is understood
- API impact is understood
- Privacy and data integrity impact is understood
